# flood_pso_viewer 開発記録

> 起点: 2026-04-30
> 目的: `flood_pso/` の NBT 出力を **ネイティブ GPU で** 閲覧するビューアの実装。
> Web 版（`schematic-renderer` 等）が大規模 NBT でクラッシュしたため、
> 16-32GB メモリ機でも回せる Rust ネイティブ版が必要。

## 設計方針

| 軸 | 選択 | 理由 |
|---|---|---|
| 言語 | **Rust** | 高負荷耐性、ゼロコスト抽象、wgpu アクセス |
| ゲームエンジン | **Bevy 0.15** | wgpu/winit のお膳立て、ECS、PBR が標準 |
| 入力 NBT 形式 | Minecraft Structure NBT (gzip) | `flood_pso/nbt_export.py` の出力に一致 |
| パース | **fastnbt** | 高速・人気・serde 統合 |
| メッシュ生成 | **自作 greedy mesher**（rayon 並列） | `mesh_builder_wasm` を参考に書き直し |
| カメラ | bevy_panorbit_camera | 標準的な軌道カメラ |
| UI | bevy_egui | side panel で flood_pso_meta を表示 |

`schematic-renderer/mesh_builder_wasm` のフルポートは検討したが、
WASM bindings + js-sys + テクスチャ/UV 周りで 1,288 行が絡み合っており、
**flood_pso 用途（マテリアル数 8 / 単色描画）には過剰**。
アルゴリズム設計だけ参考にして、専用シンプル版を新規実装。

## ファイル構成

```
src/
├── main.rs          # Bevy App、CLI（clap）、シーン構築
├── voxel.rs         # Material enum、VoxelGrid（dense + bbox 追跡）
├── nbt_loader.rs    # fastnbt → VoxelGrid + FloodPsoMeta 抽出
├── greedy_mesh.rs   # bbox 限定 + rayon 並列 greedy meshing
├── render.rs        # MeshBuffer → Bevy Mesh + StandardMaterial
└── ui.rs            # bevy_egui の右パネル + dh_map ヒートマップ
docs/
└── 00_開発記録.md
```

## 高速化メモ

### bbox 追跡

dense grid は md_5m で 380M セル。`filled_bbox()` を毎回フル走査すると遅いので
`set()` 中に min/max を追跡し O(1) 取得。

### slice 並列化

greedy meshing は 6 面 × N slice の二重ループで構成。
slice 同士は独立なので `into_par_iter().map(...)` で並列化。
slice ごとに per-material `MeshBuffer` を作って後で merge。

### 結果

K=16 md_5m:
- NBT load: 0.54s
- greedy mesh: **2.71s**（並列化前は 12s+ で timeout）
- 出力: 10.5M verts, 2.6M quads

## 既知の制約と次の改善案

1. **dense grid のメモリ**: huge_5m (15km × 15km @ 5m) で 3000 × ny × 3000 ≈ 数GB。
   chunk-based sparse 表現が必要。
2. **テクスチャ未対応**: マテリアル別単色のみ。Minecraft resource pack を読みたい場合は
   `schematic-renderer` の `ResourcePackManager` 系を参考に実装。
3. **シャドウ未対応**: 大規模 quad 数で重いため OFF。後で cascaded shadow を追加可能。
4. **wayland 対応外**: WSL2 で wayland-client が無くビルド失敗するため `x11` のみ。
   ネイティブ Linux で wayland 使いたい場合は Cargo.toml の features に `wayland` 追加。
